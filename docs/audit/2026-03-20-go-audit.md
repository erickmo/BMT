#### [C-001] Jurnal Keuangan Error Di-ignore — Inkonsistensi Akuntansi

- **File:** `internal/service/rekening_service.go`
- **Baris:** ~80, ~128
- **Masalah:** Error dari `PostJurnal` di-buang dengan `_`. Jika posting jurnal gagal, transaksi tetap dianggap sukses, menghasilkan perubahan saldo rekening tanpa catatan jurnal double-entry.
- **Dampak:** **Kritis** — Inkonsistensi akuntansi yang tidak terdeteksi. Saldo berubah tapi tidak ada jejak jurnal. Melanggar prinsip utama "SETIAP TRANSAKSI KEUANGAN = jurnal".

```go
// ❌ Sebelum — error di-ignore
_ = s.jurnalService.PostJurnal(ctx, PostJurnalInput{...})

// ✅ Sesudah — error harus menghentikan transaksi
if err := s.jurnalService.PostJurnal(ctx, PostJurnalInput{...}); err != nil {
    return fmt.Errorf("gagal posting jurnal: %w", err)
}
```

#### [C-002] Setor/Tarik Tidak Dalam DB Transaction — Risiko Partial Failure

- **File:** `internal/service/rekening_service.go`
- **Baris:** Fungsi `Setor` dan `Tarik`
- **Masalah:** `UpdateSaldo` dan `CreateTransaksi` dipanggil sebagai dua query terpisah tanpa `pgx.Tx`. Jika `CreateTransaksi` gagal setelah `UpdateSaldo` berhasil, saldo berubah tapi tidak ada catatan transaksi.
- **Dampak:** **Kritis** — Data inconsistency permanen. Saldo nasabah berubah tanpa riwayat transaksi.

```go
// ❌ Sebelum — dua query terpisah, tidak atomik

s.transaksiRepo.CreateTransaksi(ctx, tx)

// ✅ Sesudah — semua dalam satu DB transaction
s.db.WithTx(ctx, func(tx pgx.Tx) error {
    // 1. SELECT rekening FOR UPDATE
    // 2. Validasi domain
    // 3. UpdateSaldo (dengan tx)
    // 4. CreateTransaksi (dengan tx)
    // 5. PostJurnal (dengan tx)
    // 6. CatatUsageLog (dengan tx)
    return nil

```

#### [W-001] `rows.Err()` Tidak Dicek Setelah Loop — Silent Data Truncation

- **File:** `internal/repository/postgres/autodebet_repo.go`, `rekening_repo.go`
- **Baris:** Semua fungsi yang menggunakan `rows.Next()` loop (ListConfigByRekening, ListConfigAktifByBMT, ListJadwalByTanggal, ListTunggakanByRekening, dll.)
- **Masalah:** Setelah loop `for rows.Next()`, tidak ada `if err := rows.Err(); err != nil { return nil, err }`. Jika koneksi DB putus di tengah iteration, hasil yang dikembalikan bisa tidak lengkap tanpa error.

```go
// ✅ Tambahkan setelah setiap loop
for rows.Next() { ... }
if err := rows.Err(); err != nil {
    return nil, fmt.Errorf("gagal iterasi rows: %w", err)
}
```

## Arsitektur — Poin Positif

Beberapa hal yang sudah baik dan harus dipertahankan:

1. **Clean Architecture** — Layer handler → service → repository dipatuhi dengan baik. Interface didefinisikan di domain layer.
2. **Domain Errors** — Error sentinel terdefinisi di setiap domain (`ErrSaldoTidakCukup`, `ErrRekeningBeku`, dll.) — tersentralisasi dan konsisten.
3. **`money.Money` type (int64)** — Tidak ada float untuk nilai uang. Konsisten di seluruh codebase termasuk di repository (Int64(), money.New()).
4. **Tenant Isolation** — Query repository selalu menyertakan `bmt_id`. Filter `WHERE bmt_id = $1 AND is_aktif = true` konsisten.
5. **Parameterized Query** — Tidak ada raw SQL interpolation. Semua query menggunakan `$1, $2, ...`.
6. **`defer rows.Close()`** — Selalu ada setelah setiap `Query()` call.
7. **Domain Logic di Domain Layer** — `ValidasiTransaksi`, `TutupSesi`, `EksekusiAutodebet`, `Ajukan` — semua business logic di domain, bukan di handler atau repository.
8. **Idempotency Key** — Domain NFC sudah mendukung idempotency key dengan `GetTransaksiByIdempotency`.
9. **NFC PIN tidak pernah di-expose** — `PINHash string \`json:"-"\`` — field sensitif tidak masuk JSON response.
